# Externalize — 把自动加载 Skill 变成按需外挂

> 不是标准化商品，是你自己上手的半成品。

## 这是什么

externalize 帮你把 `.agents/skills/` 中的自动加载技能转化为：
- **AGENTS.md 触发条目**（你自定义的关键词，Agent 精确匹配后加载）
- **`docs/workflows/{name}.md` 实现文件**（动作流程，按需读取）

结果：常驻上下文的 skill 描述 → 一行触发词。30 个技能的外部化可以节省 ~35K token 的上下文占用。

## 为什么这样设计

市场上流行的 skill 系统靠**语义描述自动匹配**——Agent 从描述中推断"用户现在可能需要这个技能"。这个模式适合给百万人用的产品：用户不知道系统有什么能力，需要自动发现。

但这是你自己的工具。你知道你装了 add-todo，你知道什么时候该用。你不需要 Agent 替你发现——你需要它**快速响应**。

externalize 放弃语义推断，选择精确关键词触发。消除不必要的推理层，把上下文还给真正需要思考的任务。

## 你可以做的事

- **调触发词**。原 skill 叫 `add-todo`，你说"记一笔"更顺手？改。你说"后面做"而不是"回头处理"？改。你的触发词，你的习惯。
- **拆技能**。看到别人写的 skill 觉得思路好但不想让太多技能挤占上下文？拆开它，让 Agent 的启动上下文只剩一行触发条件。需要时才按行加载，不占常驻空间。
- **造流程**。新建模式里，你描述一个需要 AI 执行的能力，externalize 采访你收敛需求，直接产出你的触发词 + 你的工作流。

externalize 的角色是**帮你拆、帮你生成**，不是替你定格式。最终留在你项目里的触发词和 workflow，都是你亲手调过的。

## 快速开始

```bash
curl -o .claude/skills/externalize/SKILL.md --create-dirs <仓库URL>
```

装好后，对 AI 说：

- **"把 xxx 外部化"** → 转化已有 skill
- **"把这个外部化 https://raw.../some-skill/SKILL.md"**→ 转化看好的 skill
- **"我想新建一个能力，用来 xxx"** → 从零定义新 workflow
- **"删除 xxx 外部化"** → 移除触发词和工作流，归档去留你来定

AI 会自己处理拆解、生成、归档。你不用碰命令行。

## 三种模式

| 模式 | 做什么 | 谁执行 |
|------|--------|--------|
| **转化** `convert` | 读取已有 SKILL.md → 拆出触发词 + 动作 → 写入 AGENTS.md + workflow | 脚本 |
| **新建** `new` | 采访用户收敛需求 → 直接产出 externalize 格式 | AI（通过 externalize SKILL.md） |
| **移除** `remove` | 删除触发词 + workflow，交互式决定归档去留 | 脚本 |

## 对“externalize转化或新建的技能”的触发机制

AGENTS.md 中的条目：

```markdown
- {触发词|变体1|变体2} → `docs/workflows/{name}.md`
```

Agent 扫描时精确匹配任一触发词即加载对应 workflow。加载后 workflow 内置 y/n 确认机制，防止误触发。

## 什么技能该外部化、什么不该

| 该保持自动加载 | 该外部化 |
|---------------|---------|
| 极常用，每轮都碰得到 | 明确知道何时该用，触发词喊得出来 |
| 触发条件边界模糊，靠 Agent 语义推断 | 别人推荐的、博主说好用但你不确定值不值得 |

> 原则：不给未验证的工具付上下文租金。拿不准的技能先外部化再试用——不好用就删，一个命令清理干净，归档去留由你决定。
