### ADR-010: init-skeleton.py 从本地路径改为 GitHub 拉取模式 [来源:母库 @2026-06-01]

| 字段 | 内容 |
|------|------|
| **状态** | Active |
| **日期** | 2026-06-01 |
| **问题** | `init-skeleton.py` 需要 `--skeleton` 参数或 `SOP_SKELETON_PATH` 环境变量指向本地骨架路径，但不同设备路径不同，跨设备体验差，新用户困惑 |
| **候选方案** | A. 保持本地路径模式，增加更多错误提示<br>B. 改为从 GitHub 拉取，去掉 `--skeleton` 本地回退（纯 GitHub 模式）<br>C. GitHub 拉取 + `--skeleton` 本地回退双模式 |
| **决策** | **B. 纯 GitHub 拉取模式** |
| **理由** | 1. `init-skeleton.py` 的目标用户是空白新项目，没有自定义骨架需求<br>2. 已有项目的 init 可通过 `starter/` 手动复制或 AI 触发词解决<br>3. 所有拉取的文件都是公开仓库的纯文本，网络成本极低<br>4. 零配置即可跨设备使用 |
| **后果** | 正面：用户只需 `curl + python init-skeleton.py`，无需任何路径配置。负面：离线环境不能使用。 |
| **可逆性** | 高。恢复 `--skeleton` 参数并改为双模式即可回退。 |

### ADR-011: 阶段产出不放 starter 模板 [来源:母库 @2026-06-10]

| 字段 | 内容 |
|------|------|
| **状态** | Active |
| **日期** | 2026-06-10 |
| **问题** | 初始为 `docs/frontend.md`（阶段二阶段产出）创建了 `starter/docs/frontend.md` 模板，但骨架原有约定是阶段产出均由 AI 按需生成，starter/ 只放基础设施和自检工具 |
| **候选方案** | A. 保留模板：frontend.md 作为阶段二可选产出放到 starter/，项目按需复制<br>B. 删除模板：遵守骨架约定，在 workflow 中用结构化描述定义 frontend.md 的标准章节 |
| **决策** | B. 删除模板，用 workflow 结构化描述驱动 |
| **理由** | 1. stage outputs 高度依赖项目上下文，模板指导价值不如结构化描述高<br>2. 模板与 workflow 描述不一致时产生额外维护成本<br>3. 已有反例：design.md、brief.md、database.md 均为阶段产出且无 starter 模板——frontend.md 不应例外<br>4. workflow 中的「AI 行为规范」可直接约束 AI 的产出格式和行为 |
| **后果** | 正面：保持骨架约定一致性；减少一个需要维护的模板文件；AI 生成的内容更贴合项目上下文。负面：无。 |
| **可逆性** | 高。重建 `starter/docs/frontend.md` 并更新 workflow 即可。 |
